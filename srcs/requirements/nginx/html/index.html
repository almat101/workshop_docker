Cos'è Docker?

La piattaforma per il Software Packaging

🚀 Performance

Feature VM Docker
OS Guest OS completo Kernel condiviso
Peso Gigabytes Megabytes
Boot Minuti Millisecondi

📦 Isolamento

  • 🛡️ Processo: Ogni container vede solo i suoi processi (namespace PID separato).
  • 🌐 Rete: Ogni container ha la sua scheda di rete virtuale (IP e porte propri).
  • 📁 File System: Il container vede solo i file dell’immagine e dei suoi volumi.

🌍 Portabilità

"Build once, run anywhere". Lo standard universale che garantisce lo stesso comportamento su ogni macchina.

Immagine vs Container

Dalla "Ricetta" al "Piatto Servito"

1. Il Dockerfile

                        FROM alpine:3.22
                            
RUN apk update && apk add nginx

COPY conf /etc/nginx

COPY html /var/www/html

CMD ["nginx","-g","daemon off;"]

La Ricetta: un file di testo con le istruzioni.

2. L'Immagine

💾

Lo Snapshot statico.

  • Buildata dal Dockerfile.
  • Read-Only I file di base non cambiano mai.
  • Contiene OS + App + Config.

3. Il Container

⚙️

L'Istanza Viva.

  • L'immagine in esecuzione.
  • Read-Write layer: I cambiamenti sono temporanei..
  • PID 1: Il container vive finché vive il processo principale.

Comandi di Sopravvivenza

Il toolkit essenziale per la gestione dei container

docker build -t nome_immagine .

Builda la "ricetta" (Dockerfile) e crea l'immagine statica.

docker run nome_immagine

Crea e avvia un container basato sull'immagine scelta.

docker ps [-a]

Lista i container attivi [o tutti, inclusi quelli spenti].

docker logs [nome_container]

Visualizza i log/output del container: utile per debug, errori.

docker exec -it [nome_container] sh

Apre una shell (sh) nel container.

Laboratorio: Build & Run

Dalla teoria alla pratica: il mistero dell'isolamento

# 1. Costruiamo l'immagine

$ docker build -t my-nginx .

📦 Immagine pronta:
Contiene Alpine + Nginx + i tuoi file (COPY).

# 2. Avviamo il container

$ docker run -d --name web-1 my-nginx

Domanda: secondo voi ora funziona❓

Verificate su localhost nel browser...

🕵️ Test 1: lanciate
docker exec web-1 curl localhost
Cosa ti risponde il container?

🕵️ Test 2: lanciate
docker ps.
Analizza l'output: riesci a trovare cosa manca?

# 3. Creiamo il "Ponte" (Port Mapping)

$ docker run -d -p 8081:80 --name web-2 my-nginx

✅ Successo!
Il traffico ora passa dall'Host (8081) al Container (80).

🚀 In Inception automatizzeremo tutto questo con il Docker Compose.

Perché Docker Compose?

Quando un container non basta più

❌ Senza Compose

  • 🖥️ 3 container = 3+ comandi
  • 🔀 Ordine di avvio manuale
  • 🧠 Devi ricordarti tutto
  • 💥 Fragile, non riproducibile

✅ Con Compose

  • 📄 Un solo file: docker-compose.yml
  • 🔁 Ordine e dipendenze automatiche
  • 🔧 Configurazione dichiarativa
  • ♻️ Sempre lo stesso risultato

Docker-compose coordina più container Docker come un’unica applicazione.

👉 Inception inizia qui: più servizi, una sola infrastruttura.

Il Trio di Inception

Oltre il web server: i motori dell'infrastruttura

🐘 PHP-FPM

Nginx non sa "leggere" il PHP, sa solo consegnare file statici.

PHP-FPM è il motore che esegue il codice di WordPress e restituisce il risultato a Nginx.

🔍 Ascolta sulla porta 9000.

💾 MariaDB

È la memoria del tuo sito. Qui vengono salvati post, commenti, utenti e configurazioni.

Senza un Volume, ogni riavvio del container MariaDB cancellerebbe tutto il sito.

🔍 Ascolta sulla porta 3306.

docker-compose.yml

Un file per descrivere tutta l’infrastruttura 

            
            services:
              nginx:
                build: ./requirements/nginx
              ports:
                - "443:443"
              restart: always
              volumes:
              - wordpress:/var/www/html
              networks:
                - inception

            volumes:
              wordpress:
                
            networks:
              inception:
                driver: bridge
  • 📦 services = container
  • 🛠️ build = Dockerfile
  • 🔌 ports = mappatura porte
  • 🔄 Restart: Gestione automatica dei crash.
  • 💾 volumes = persistenza dati
  • 🌐 networks = isolamento e comunicazione

👉 Inception è questo file + regole precise.

Docker Network

Perché localhost non è tuo amico

❌ Errore comune

  • 🌐 Usare localhost tra container.
  • 🔗 Cercare di indovinare l'IP.
  • 💥 Il sistema è fragile e isolato.

✅ Docker DNS

  • 🕸️ Rete Bridge condivisa.
  • 📛 Ogni container ha un nome DNS.
  • 🔄 Comunicano tramite Service Name.

In Inception, il collegamento avviene così:
WordPress → mariadb:3306
Nginx → wordpress:9000

👉 Ogni container vede se stesso come "localhost".
La rete Docker è il tunnel che li unisce.

Docker Volumes

I container muoiono. I dati no.

❌ Senza volume

  • 💥 Dati persi ogni volta
  • 🔄 Rebuild = reset
  • 😱 Inception fallito

✅ Con volume

  • 💾 Dati persistenti
  • ♻️ Container sacrificabili
  • 🧠 Architettura corretta

In Inception:
📦 MariaDB → volume
📁 WordPress files → volume

👉 I container sono stateless. La persistenza vive fuori...

Dove vivono i dati?

Collegare il container alla macchina Host (VM)

Named Volume (Standard)

            volumes:
  - wordpress:

Gestito da Docker in:
/var/lib/docker/volumes/...
Semplice, ma percorso "nascosto".

Named + Bind (Inception)

            volumes:
  wordpress:
    driver: local
    driver_opts:
      type: none
      device: /home/amatta/data/wp
      o: bind

Requisito Project: puntiamo
alla nostra cartella /home
tramite l'opzione device.

💡 Usiamo un Named Volume che punta a un Bind Mount per persistenza e controllo.

Env vs Secrets

Configurazione ≠ Segreti

.env (Configurazione)

            DOMAIN_NAME=amatta.42.fr
            MYSQL_USER=wp_user
            MYSQL_DATABASE=maria_user

Dati necessari al setup.
Visibili nel sistema con env.

Secrets (Credenziali)

            db_password.txt
            wp_password.txt
            credentials.txt

Dati sensibili.
Montati come file in /run/secrets/.

❌ Password hardcodate nei file
❌ Password su Git
➡ Progetto fallito

👉 Inception valuta sicurezza e separazione delle responsabilità.
